### 二、已知 Bug / 邏輯錯誤（❌ 壞的）

#### B1. 月份 padding 邏輯錯誤（PadAfter 永遠只有 1 天）
**位置**：`CalendarView.tsx:65-70`

```63:70:src/components/CalendarView.tsx
    // Pad end to complete the last week
    const endDay = end.getDay();
    const padAfter = Array.from({ length: (7 - endDay - 1) % 7 + 1 }, (_, i) => {
      const d = new Date(end);
      d.setDate(d.getDate() + i + 1);
      return d;
    });
```
**問題**：當 `endDay = 6`（週六）時，`(7-6-1) = 0 % 7 = 0`，`0+1 = 1`，永遠只 pad 1 天；當 `endDay = 5`（週五）時，`(7-5-1) = 1`，正確 pad 2 天。這導致某些月份的日曆網格行數不足，月曆右側不完整。

**正確演算法**：要 pad 到「下一週的週日」，即 `padAfter` 長度 = `(7 - endDay) % 7`（若等於 0 表示正好填滿一週，不需要 pad）。

---

#### B2. 格內計數只算未完成（誤導：以為這天沒任務）

```216:src/components/CalendarView.tsx
            const taskCount = dayTasks.filter((t) => t.status !== "done").length;
```

### 三、功能缺口（△ 未實作，但用戶可能需要）

#### M1. 月/週視圖切換
日曆只有月視圖，沒有週視圖。週視圖可以看到更多任務細節，是 TickTick/Google Calendar 的標配。

#### M2. 格子內顯示任務標題（前綴截斷）
理想 UX：每個格子最多顯示 1-2 個任務名稱（超出的用 +N 暗示），讓用戶一眼看出「這天有什麼」。這解決 B3 的根本問題。

#### M3. 逾期任務明顯標記
遲到的任務（`dueDate < 今天` 且 `status !== "done"`）應在格子內以紅色字或紅色邊框標記，而不是只靠優先級色。

#### M4. 子任務進度在格子內顯示
`task.subTasks` 若有已完成/未完成的子任務，格子內應顯示進度（如 `[2/5]`），讓用戶評估任務完成度。

#### M5. 重複任務識別
有 `recurrence` 的任務，格子應有小圖示標記（如 🔁）。

#### M6. 日曆進場默認行為
進入日曆視圖時，沒有自動選中今天並展開任務列表。用戶必須手動點擊日期。

#### M7. 月份年份跳轉
沒有「年月選擇器」（如點擊月份名稱彈出picker），只能一步一步 prev/next。

#### M8. 任務列表的排序
`selectedDateTasks` 沒有明確排序：優先級 → 建立時間 → 字母？用戶看到列表順序可能感到隨機。

#### M9. 響應式日曆格子
`height: calc(72px * 6)` 固定高度，在小手機上（320px 寬）每格只有 ~45px 寬，數字都擠在一起。

#### M10. 週視圖的「每日時段」或「滾動流」
若要做週視圖，理想上是左側時間軸（00:00-23:00），右側每天一列，任務按時段定位。本專案 `dueTime` 只用於顯示，沒有時段定位。

#### M11. 多清單/標籤過濾
日曆視圖沒有按清單或標籤過濾的能力，所有任務混在一起。

#### M12. 移動端拖放
拖放功能在觸控設備上基本不可用（沒有 touch 事件）。

## 核心 UX 原則

1. **Immediate Revelation** — 資訊優先可見，減少點擊
   - 子任務預設展開，不需要展開/折疊操作
   - 點擊任務卡片 → 開啟編輯面板

2. **Smart Defaults** — 智慧分類減少手動操作
   - Today / Upcoming / Flagged 等自動過濾視圖

3. **Visual Hierarchy** — 清晰的視覺層級
   - 已完成任務：灰階 + 刪除線
   - 任務間距充足，呼吸感強
