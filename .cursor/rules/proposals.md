## 通過評分、待生效的草案

### Proposal 16 — 自我 UAT 紀律（AI 自驗收 Gate）
**評分：9.0**（觸發觀察：tag 修復 commit 後未在瀏覽器實測，浪費你的 UAT 時間）
**期望生效條文：**
> 當 AI 完成任何 commit 並 push 前，必須用 `browser_navigate` +
> `browser_snapshot` 在本地 dev server 自我走一次主要互動（例如標籤頁的
> 「新增 → 出現 → 刪除 → 消失」），確認行為正確才宣佈完成。
> - 若 sandbox / dev server 不可用，必須在 commit message 結尾明示
>   `⚠️ unable to self-verify: <原因>`，不假裝驗過。
> - 例外：純文字 / 註解 / 1-2 行 typo 修正可跳過。

---

### Proposal 17 — 評分前必讀 Codebase（Reuse-First Gate）
**評分：9.0**（觸發觀察：tag 修復浪費一輪才發現 `storage.ts` 早就有 `getTags/saveTags` 函數）
**期望生效條文：**
> 提出任何方案評分前，AI 必須先用 `Grep/Glob/Read` 掃描是否已有可複用資源：
> - 類似的 storage key / state hook / utility 函數
> - 已存在的資料流路徑（避免重複造輪子）
> - 命名空間衝突（例如本次發現兩套「tag」系統並存）
>
> 發現既有架構可直接複用時，**必須在評分表中標明**
> 「發現既有 X 可複用」，再展開方案；如不可複用，說明原因。
> 否則視為評分不完整、等同違憲。

---

### Proposal 18 — 修憲觸發條件（Reflection Trigger）
**評分：8.0**（觸發觀察：「累計浪費」只在使用者抱怨後才被發現，無自我檢視節點）
**期望生效條文：**
> 每次完成階段性 commit 前，AI 必須主動評估本輪是否出現下列任一現象：
> 1. 方案反覆 ≥ 2 次（先做 A → 改 B → 再回 A）
> 2. 評分表被推翻（使用者回「換 X」後又回原方案）
> 3. 走回頭路（修改後 revert / 復原前一版）
> 4. 發現既有架構可複用卻浪費一輪才發現（Proposal 17 觸發）
>
> 任一為真 → commit message 結尾追加「🔧 修憲建議：<條文草案>」，
> 不直接修憲，由使用者判定是否寫入 proposals.md / global.mdc。
